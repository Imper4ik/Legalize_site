## Ключевые закрытые проблемы (история трёх аудитов)

1. **OCR-ревью только Admin/Manager + флаг `can_run_ocr_review`** — Staff
   убран из `OCR_REVIEW_ALLOWED_ROLES`; кнопки в UI скрыты через
   context processor `staff_capabilities`.
2. **Мульти-кейсовые клиенты не роняют MOS-ревью** — явный `?case=<uuid>`,
   выбор дела при неоднозначности (`mos_review_select_case.html`).
3. **Документы**: выдача файлов архивных дел, явный выбор дела при загрузке,
   наследование флагов test/demo-данных.
4. **Почта/уведомления**: case-scoped идемпотентность, отказ от отправки при
   неоднозначном деле вместо угадывания; автономный guard не шлёт реальные
   письма для test/demo-данных.
5. **i18n**: ~252 исправления на язык; ru-каталог заполнен (иначе Django
   падает в польский, а не в msgid); плюрали исправлены.
6. **Статика вендорена** (`static/vendor/`), CDN-хосты удалены из CSP —
   `script-src 'self'` (nonce для inline), `style-src` без CDN.
7. **GDPR-ретеншн подключён к расписанию**: `/cron/run-maintenance/`
   (cleanup_email_logs + anonymize_old_clients) и ежедневный шаг в
   `run_background_automation_loop`; анонимизация — dry-run, пока не включён
   `AUTO_ANONYMIZE_OLD_CLIENTS` (см. `docs/deploy.md`).
8. **Клиентский i18n и доступность**: RODO/личные данные полностью переведены
   на PL/EN/RU, чистая установка заполняет локализованные основания подачи,
   `<html lang>` корректен, основные поля онбординга связаны с подписями.
9. **Приватность QR**: ссылка онбординга кодируется только локальной библиотекой
   и больше не передаётся стороннему QR API.

## Принятые и задокументированные компромиссы

- **Поиск по открытым полям** — имя/фамилия/email/телефон клиента не
  шифруются ради поиска; см. `docs/field_encryption.md`.
- **«Еженедельные» проверки** реализованы как ежедневные прогоны с недельной
  дедупликацией отправки; см. `docs/BUSINESS_WORKFLOW.md`.
- **`/cron/db-backup/`** остаётся на внешнем планировщике (нужен `pg_dump`);
  всё остальное покрывает background-loop.
