## Ключевые закрытые проблемы (история трёх аудитов)

1. **OCR-ревью только Admin/Manager + флаг `can_run_ocr_review`** — Staff
   убран из `OCR_REVIEW_ALLOWED_ROLES`; кнопки в UI скрыты через
   context processor `staff_capabilities`.
2. **Мульти-кейсовые клиенты не роняют MOS-ревью** — явный `?case=<uuid>`,
   выбор дела при неоднозначности (`mos_review_select_case.html`).
3. **Документы**: выдача файлов архивных дел, явный выбор дела при загрузке,
   наследование флагов test/demo-данных.
4. **Почта/уведомления**: case-scoped идемпотентность, отказ от отправки при
   неоднозначном деле вместо угадывания; автономный guard не шлёт реальные
   письма для test/demo-данных.
5. **i18n**: ~252 исправления на язык; ru-каталог заполнен (иначе Django
   падает в польский, а не в msgid); плюрали исправлены.
6. **Статика вендорена** (`static/vendor/`), CDN-хосты удалены из CSP —
   `script-src 'self'` (nonce для inline), `style-src` без CDN.
7. **GDPR-ретеншн подключён к расписанию**: `/cron/run-maintenance/`
   (cleanup_email_logs + anonymize_old_clients) и ежедневный шаг в
   `run_background_automation_loop`; анонимизация — dry-run, пока не включён
   `AUTO_ANONYMIZE_OLD_CLIENTS` (см. `docs/deploy.md`).
8. **Клиентский i18n и доступность**: RODO/личные данные полностью переведены
   на PL/EN/RU, чистая установка заполняет локализованные основания подачи,
   `<html lang>` корректен, основные поля онбординга связаны с подписями.
9. **Приватность QR**: ссылка онбординга кодируется только локальной библиотекой
   и больше не передаётся стороннему QR API.

## Модель ролей: владелец vs главный сотрудник (обновление 2026-07)

Роль **Manager** теперь — «главный сотрудник», а не совладелец. Управление
персоналом и ролями (`staff_manage`/`role_manage`), фирменные/юридические и
RODO-настройки (`AppSettings`: юрлицо, NIP/REGON/KRS, политика приватности,
идентичность администратора данных), шаблоны документов, основания подачи и
прайс (`service_price`) переведены в разряд **owner-only** — доступ только у
роли `Admin`/суперпользователя (`SETTINGS_ALLOWED_ROLES`,
`CRITICAL_SETTINGS_ALLOWED_ROLES`, `PEOPLE_ALLOWED_ROLES = ("Admin",)`).

Закрытые этим изменением риски:
- менеджер больше не может выдать себе (или другому) роль `Admin` и получить
  права владельца;
- менеджер не может редактировать/деактивировать аккаунт владельца.

Дополнительный уровень защиты в `staff_manage_view`/`StaffUser*Form`: только
суперпользователь может выдавать группу `Admin`, и только суперпользователь
может редактировать или менять статус аккаунтов уровня владельца (суперюзер
или группа `Admin`) — на случай, если в конфигурации ролей снова появится
не-суперпользовательский администратор. Операционные возможности менеджера
(клиенты, документы, платежи, задачи, письма, отчёты, логи, OCR-review)
сохранены полностью. Покрыто тестами
`clients/tests/test_staff_management_hardening.py` и
`test_role_permissions_stage14.py::*manager*`.

## Автономность (обновление 2026-07)

Фоновый loop сделан более самодостаточным:

- **Самопланируемый бэкап БД.** `ENABLE_INPROCESS_DB_BACKUP=true` — и loop сам
  делает `pg_dump` раз в сутки (дневной гейт ставится только после успешного
  бэкапа, поэтому сбой ретраится на следующем цикле), с алертом админам при
  ошибке. Убирает единственную обязательную внешнюю зависимость
  (`/cron/db-backup/`); для надёжности комбинируется с
  `BACKUP_REMOTE_STORAGE=true` (S3/R2/B2).
- **Сторож простоя.** Долгоживущий маркер `last_run` позволяет заметить, что
  loop был недоступен, и при возобновлении отправить админам письмо
  «автоматизация простаивала ~N минут». Дополняет уже существующую отдачу 503
  на `/readyz` при протухшем heartbeat.

Покрыто тестами в `clients/tests/test_priority_12_13_workflow.py`
(`*daily_backup*`, `*backup_failure*`, `*resuming_after_outage*`).
