# Управление переводами и бизнес-текстами

В проекте Legalize_site используется гибридная система переводов, разделяющая интерфейсные строки (UI) и бизнес-данные (чеклисты, основания подачи).

## 1. UI-переводы (Интерфейс)

UI-переводы — это кнопки, заголовки, сообщения об ошибках, метки полей и другие элементы интерфейса, которые обёрнуты в коде в `_()` или `{% translate %}`.

### Хранение
- **Fallback (Источник по умолчанию)**: Файлы локалей `locale/*/LC_MESSAGES/django.po`. Они хранятся в Git.
- **Основной источник (Production)**: Таблица `TranslationOverride` в базе данных PostgreSQL на Railway.

### Как это работает
1. При отображении страницы Django ищет перевод в `.po/.mo` файлах.
2. Если включен режим `TRANSLATION_DB_OVERRIDES_ENABLED=True`, система проверяет наличие активного перевода в таблице `TranslationOverride` для текущего языка и `msgid`.
3. Если перевод в БД найден — используется он. Если нет — используется стандартный перевод из файлов.

### Translation Studio
Translation Studio позволяет редактировать переводы прямо из интерфейса. В зависимости от настройки `TRANSLATION_STUDIO_STORAGE` в `.env`:
- `database` (по умолчанию): Сохраняет изменения в PostgreSQL. Изменения не пропадают при перезапуске контейнера на Railway.
- `po`: Сохраняет изменения только в `.po` файлы (не рекомендуется для Railway).
- `both`: Сохраняет и в БД, и в `.po`.

## 2. Экспорт и Импорт UI-переводов

Для синхронизации переводов между БД и Git используются management-команды.

### Экспорт из БД в файлы `.po`
Перед тем как делать релиз или коммитить изменения в Git, выгрузите переводы из БД в файлы:
```bash
python manage.py export_db_translations_to_po
```
Команда обновит `.po` файлы и автоматически скомпилирует `.mo` файлы. Если вы не хотите компилировать, добавьте флаг `--no-compile`.
Для проверки без сохранения используйте `--dry-run`.

### Импорт из файлов `.po` в БД
Если вы добавили новые переводы в файлы и хотите загрузить их в БД на Railway:
```bash
python manage.py import_po_to_db
```
По умолчанию команда НЕ перезаписывает существующие в БД переводы. Чтобы принудительно обновить их из файлов, используйте:
```bash
python manage.py import_po_to_db --overwrite
```
Для проверки используйте `--dry-run`.

## 3. Бизнес-значения (Контент)

Бизнес-значения — это данные, которые определяют логику работы и контент сайта. Они **НЕ** являются UI-переводами и управляются отдельно.

- **Основания подачи**: Редактируются через Django Admin в модели `Submission` (поля `name_pl`, `name_en`, `name_ru`).
- **Чеклисты и названия документов**: Редактируются через Django Admin в модели `DocumentRequirement` (поля `custom_name_pl`, `custom_name_en`, `custom_name_ru`).
- **Цены**: Редактируются через Django Admin в модели `ServicePrice` (если применимо).
- **Настройки фирмы**: Редактируются через `AppSettings`.

Эти данные хранятся **исключительно в базе данных** и никогда не сбрасываются при деплое. Для них не нужно использовать Студию Перевода.

## Translation Studio UI language

Translation Studio is an internal staff/developer tool. Its management UI may remain English-only unless a product requirement explicitly asks for localized staff tooling. Public/customer-facing strings should still use Django i18n and translation overrides.
