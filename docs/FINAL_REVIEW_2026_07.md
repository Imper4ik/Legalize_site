## 2. Найдено и исправлено в этом прогоне

**«Панель администратора» (`/staff/admin-dashboard/`) рендерилась пустой.**
Шаблон `clients/admin_dashboard.html` ожидает развёрнутый контекст
(статусы email/OCR, диск, зависимости, операционные очереди, недавние
рассылки), а `AdminDashboardView` передавал три переменные, ни одна из
которых шаблоном не используется. Django молча подставлял пустые строки —
страница выглядела «свёрстанной, но без данных». Существующие тесты
проверяли только код ответа 200 и дефект не ловили.

Исправление: view собирает полный контекст из уже существующих сервисов
(`legalize_site.runtime.runtime_dependency_summary`, счётчики
`DocumentProcessingJob`/`EmailCampaign`/`Document`/`StaffTask`/`Reminder`,
агрегат `DatabaseMediaFile` для диска). Добавлен регрессионный тест,
который проверяет наличие всех переменных контекста
(`test_admin_dashboard_context_is_populated`).

Урок: тесты на `status_code == 200` не защищают от «тихо пустых» шаблонов —
см. рекомендацию про `string_if_invalid` ниже.

## 3. Оценка

### Архитектура — хорошо, зрелая для этапа
- Чистое разделение: `clients/` (домен) с выделенными `services/`, `views/`,
  `models/`, `forms/`; отдельные приложения `submissions`, `users`,
  `translations`, `database_media`, `fernet_fields`.
- Case-first модель завершена, legacy-шимы удалены, консистентность
  подкреплена моделью `consistency.resolve_required_case` и миграциями.
- Очереди (OCR, email) — идемпотентные, с lease/retry, работают и через
  in-process loop, и через cron-webhook'и: деплой без Celery/Redis-воркеров
  возможен, путь масштабирования описан.
- Graceful degradation задокументирована и реализована (OCR, email, шрифты,
  курсы валют, backup).

### Безопасность — сильная сторона проекта
- Шифрование PII на уровне полей (Fernet) с blind-index поиском; ротация
  ключей; fail-closed поведение.
- CSP без CDN (`script-src 'self'` + nonce), HSTS+preload, строгие cookie,
  rate limiting (Redis либо БД), CSRF/SSL redirect, ролевая модель
  owner/manager с owner-only настройками.
- CI: тесты (SQLite и PostgreSQL), deploy-check, ruff, mypy, bandit,
  pip-audit, дрейф миграций, i18n-проверка, shellcheck.

### Функции — полный контур CRM подтверждён E2E-прогоном
Онбординг по ссылке/QR → чек-листы документов по основаниям →
загрузка + OCR (wezwanie, ZUS RCA, паспорт) → верификация → подача →
отпечатки → решение; напоминания, массовые рассылки, платежи, семейные
дела, MOS, экспорт PDF/ZIP, RODO-ретеншн и анонимизация.

### Визуал — единый и аккуратный
- Единая тема (light/dark), Bootstrap, консистентные карточки/бейджи.
- RU/PL/EN локализация полная, включая клиентские страницы.
- Мобильная вёрстка списка клиентов с нижней навигацией — работает.
- Ни одной JS-ошибки в консоли за весь обход.

## 4. Рекомендации перед выходом на рынок

### 4.1. Быстрые (до запуска)
1. **`string_if_invalid` в тестовых настройках** — в
   `legalize_site/settings/test.py` включить
   `TEMPLATES[0]["OPTIONS"]["string_if_invalid"] = "INVALID_VAR:%s"` (или
   кастомный класс, падающий в тестах). Это ловит класс дефектов
   «шаблон ждёт переменную, view её не передал» — именно так молчала
   админ-панель. Дешёво и сразу покрывает все 160+ шаблонных тестов.
2. **Браузерный smoke в CI** — сегодняшний Playwright-обход показал себя:
   добавить в CI job (после миграций) короткий сценарий
   «логин → список клиентов → карточка → админ-панель» с проверкой
   отсутствия console.error и 4xx/5xx. 2–3 минуты на прогон, закрывает
   слепую зону между unit-тестами и продакшеном.
3. **Малваре-сканирование** — поднять clamd рядом с приложением и включить
   `MALWARE_SCAN_ENABLED=True` (единственный advisory деплой-проверки).
   Для сервиса, принимающего чужие документы, это стоит сделать до запуска.
4. **Uptime-мониторинг снаружи** — внешний пинг `/healthz/` и `/readyz/`
   (UptimeRobot/Betterstack), плюс алерт на отсутствие heartbeat
   background-loop (он уже пишется в кэш — осталось выставить наружу).

### 4.2. Автоматизация следующего уровня (после запуска)
5. **Автопроверка восстановления бэкапа** — `test_restore` уже есть;
   запланировать еженедельный прогон на staging (восстановить вчерашний
   дамп → `migrate --check` → smoke) с алертом при провале. Бэкап без
   регулярного restore-теста — это надежда, а не бэкап.
6. **Dependabot/Renovate** — pip-audit в CI ловит известные CVE, но
   обновления зависимостей сейчас ручные. Автоматические PR с обновлениями
   + зелёный CI = дешёвая гигиена.
7. **Платёжная автоматизация** — статусы платежей ведутся вручную;
   интеграция Przelewy24/Stripe (счёт → ссылка на оплату в письме →
   webhook меняет статус) уберёт ручную сверку и ускорит cash flow.
8. **Канал уведомлений помимо email** — SMS/WhatsApp для критичных
   дедлайнов (отпечатки, wezwanie): email-доставляемость в этой аудитории
   нестабильна. Архитектура очереди рассылок уже позволяет добавить второй
   транспорт.
9. **Визуальная регрессия** — скриншотные снапшоты ключевых страниц
   (Playwright `toHaveScreenshot`) на PR, чтобы CSS-правки не ломали
   вёрстку незаметно.
10. **Нагрузочный прогон по расписанию** — `generate_stress_data` уже есть;
    раз в месяц на staging: сгенерировать 5–10k клиентов, замерить p95
    списков/карточки/поиска, сравнить с базовой линией.

### 4.3. Технический долг (не блокирует запуск)
- **psycopg2-binary → psycopg 3** — Django 6 полностью поддерживает
  psycopg3; psycopg2 в maintenance-режиме. Плановая миграция.
- **Единый лимит размера страницы админ-панели**: `AdminPanelView`
  считает по всем клиентам глобально, `admin-dashboard` — тоже; при
  мультиарендности (docs/TENANT_PROVISIONING.md) счётчики надо будет
  скоупить.
- **Тип документа `statement_x` («Oświadczenie „x"»)** — отображается в
  клиентском чек-листе буквально как «Заявление „x"». Если это внутреннее
  сокращение, стоит дать документу полное имя (название правится в
  переводах требований), потому что строку видят клиенты.

## 5. Вердикт

Проект готов к пилотному запуску с первыми клиентами: прогон чистый,
деплой-проверки зелёные, безопасность выше типичного уровня MVP,
документация деплоя и go-live чек-лист актуальны. Рекомендованные пункты
4.1 стоит закрыть до активного привлечения клиентов; 4.2 — дорожная карта
автоматизации на первые месяцы эксплуатации.
