# Отчет об аудите проекта и тестовом покрытии

Этот документ содержит подробный анализ архитектуры, механизмов безопасности, кастомных системных проверок (System Checks), зависимостей и инфраструктуры тестирования проекта **Legalize CRM**.

## 1. Аудит архитектуры и механизмов безопасности

В проекте реализована многоуровневая модель защиты для работы с персональными данными клиентов (такими как гражданство, номера паспортов, файлы дел) в полном соответствии с требованиями RODO/GDPR.

### А. Безопасность на уровне Middleware (промежуточного ПО)
- **Content Security Policy (CSP)**: Управляется динамически с помощью кастомного `ContentSecurityPolicyMiddleware` в [security.py](file:///e:/Anigravity/Legalize_site/legalize_site/security.py). Считывает структурированные директивы из настроек и добавляет заголовок `Content-Security-Policy`.
- **Permissions Policy**: Кастомный `PermissionsPolicyMiddleware` прикрепляет заголовок `Permissions-Policy`, отключающий доступ к камере, микрофону, геолокации и платежным API в браузере.
- **Стандартная защита**: Использует встроенные функции безопасности Django (HSTS, защита от кликджекинга, защита от сниффинга типов данных, SSL-редирект через прокси).

### Б. Безопасность сессий и Cookie-файлов
В продакшене строгие настройки предотвращают утечку сессий и CSRF-атаки:
- `SESSION_COOKIE_SECURE = True`
- `CSRF_COOKIE_SECURE = True`
- `SESSION_COOKIE_HTTPONLY = True`
- `SESSION_COOKIE_SAMESITE = "Lax"` (можно настроить на "Strict")
- `CSRF_COOKIE_SAMESITE = "Lax"`

### В. Ограничение частоты запросов (Rate Limiting)
- **Реализация**: Кастомный `RateLimitMiddleware` ограничивает количество запросов к уязвимым эндпоинтам на основе имени URL/представления. Использует кэш-бэкенд для отслеживания лимитов.
- **Отказоустойчивость**: Параметр `RATE_LIMIT_CACHE_FAILURE_MODE` в продакшене по умолчанию равен `closed` (блокирует запросы при сбое кэша), а в среде разработки — `open` (пропускает запросы).
- **Настроенные эндпоинты**:
  - `account_login`: 5 попыток за 5 минут (по IP)
  - `account_resend_verification`: 3 запроса за 10 минут (по IP)
  - `clients:add_document`: 80 загрузок в час
  - `clients:mass_email`: 3 действия в час
  - `clients:send_custom_email`: 20 писем в час

### Г. Очистка PII (персональных данных) и логирование
- **Маскирование в Sentry**: Конфигурация Sentry отключает отправку дефолтных PII (`send_default_pii=False`) и очищает тела POST-запросов. Кастомные обработчики (`_sentry_before_send` и `_sentry_before_breadcrumb`) рекурсивно заменяют значения любых конфиденциальных полей (например, `passport`, `case_number`, `email`, `token`, `password`, `ocr`) на токен маскирования.
- **Маскирование логов**: Логгер использует `RedactPIIFilter` с регулярными выражениями для удаления телефонов, email, номеров паспортов, номеров дел и авторизационных токенов из системного вывода.

### Д. Хранение файлов и соответствие RODO
- **Приватность медиафайлов**: Загруженные файлы защищены авторизацией. Скачивание использует обфусцированные имена файлов (например, `document-123.pdf`), а не исходные имена пользователей.
- **Инфраструктура хранения**: Поддерживается объектное хранилище S3 (AWS/R2/B2), хранение файлов в базе данных (`DatabaseMediaStorage`) и постоянные Railway-тома.
- **Шифрование Fernet**: Особо чувствительные поля БД (номера паспортов, номера дел, тела писем в логах) шифруются с помощью Fernet-ключей, заданных в `FERNET_KEYS`.

## 3. Системные проверки (System Checks)

Проект содержит отличный набор кастомных системных проверок Django (`legalize_site.checks`), запускаемых на старте приложения:
1. **Настройки почты**: Выдает предупреждение, если в продакшене используется консольный бэкенд; валидирует ключи API Brevo/SendGrid.
2. **Настройки шифрования**: Проверяет, что `SECRET_KEY` и `FERNET_KEYS` заданы явно и не используют дефолтные значения.
3. **Зависимости окружения**: Проверяет наличие в системе установленных утилит `tesseract-ocr`, `pdf2image`, `poppler-utils`, `gettext` и `pg_dump`.
4. **Кэширование лимитов**: Убеждается, что для продакшена настроен распределенный кэш (Redis или Database Cache), обеспечивающий работу лимитеров запросов.
5. **Безопасность хранения**: Проверяет постоянство медиа-хранилищ в продакшене и изолирует резервные копии БД от обычных медиафайлов.
6. **Крон-токены и IP**: Контролирует наличие `CRON_TOKEN` и правила фильтрации IP для cron-задач.
7. **Консистенция схемы БД**: Сравнивает структуру моделей с реальным состоянием таблиц в БД, помогая обнаружить пропущенные миграции до выполнения запросов.

## 5. Выявление потенциальных рисков и операционных узких мест

В ходе статического анализа были выявлены следующие потенциальные риски, требующие внимания администраторов и разработчиков при масштабировании:

### А. Режим отказоустойчивости Rate Limit Middleware (`fail-closed`)
- **Риск**: В [base.py](file:///e:/Anigravity/Legalize_site/legalize_site/settings/base.py) для продакшена по умолчанию включен режим `RATE_LIMIT_CACHE_FAILURE_MODE = "closed"`. Это означает, что при падении или зависании Redis (или Database Cache) middleware заблокирует все защищаемые эндпоинты (включая вход в систему, загрузку документов и отправку email) со статусом 429.
- **Рекомендация**: Настроить надежный мониторинг здоровья кэша и перезапуска при сбоях.

### Б. Раздувание базы данных (Database Bloat) при локальном хранении медиафайлов
- **Риск**: Использование `DatabaseMediaStorage` для хранения загруженных документов непосредственно в БД удобно для MVP, но в условиях реального продакшена быстро приведет к росту таблиц, замедлению резервного копирования и снижению общей производительности БД.
- **Рекомендация**: Настроить и протестировать подключение S3/R2-совместимого хранилища с помощью `USE_S3_MEDIA_STORAGE=True`.

### В. Полное удаление тела POST-запросов в Sentry
- **Риск**: В целях защиты персональных данных (`before_send`) у всех POST-запросов полностью вырезаются данные (`request_data["data"] = REDACTION_TOKEN`). Это гарантирует отсутствие утечек PII, но может существенно усложнить диагностику ошибок, не связанных с PII (например, валидация сложных форм, сбои интеграций).
- **Рекомендация**: Разрешить логирование безопасных метаданных или нечувствительных ключей (например, статусных флагов), если это необходимо для дебага.

### Г. Чувствительность ALLOWED_HOSTS и CSRF_TRUSTED_ORIGINS к конфигурации доменов
- **Риск**: Динамический парсинг хостов в `production.py` берет домены платформ (Railway, Render) и переменные окружения. Любое обращение к сайту по альтернативному алиасу или поддомену (например, `www.example.com` вместо `example.com`), не прописанному в `ALLOWED_HOSTS`/`CSRF_TRUSTED_ORIGINS`, приведет к ошибкам 400 Bad Request или CSRF verification failure.
- **Рекомендация**: Всегда проверять соответствие DNS-записей и переменных окружения при добавлении кастомных доменов.
